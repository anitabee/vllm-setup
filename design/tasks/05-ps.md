# 05 — `ps` (runtime view + readiness)

**Type:** AFK

## What to build

The runtime view of live managed containers only, with a readiness probe.
Reuses the `managed-by` label listing from slice 03 and adds the health adapter.

For each running managed container, show: model name, host port, container
status, base URL (`http://<bind_address>:<port>/v1`), and a readiness state
obtained by polling vLLM's `/health` endpoint — distinguishing a container still
loading (`loading`) from one ready to serve (`ready`).

Scope:

- `adapters/health.py` — `httpx` GET of each model's `/health`, returning a
  readiness state; bounded, non-blocking timeouts so a still-loading model
  doesn't hang `ps`.
- `models.py` — `RuntimeContainer` (name, model, port, bind_address, container
  status, base_url, readiness), built from a Docker container plus the health
  check.
- `render.py` — the `ps` Rich table with status colors.

## Acceptance criteria

- [ ] `vllm-cli ps` lists only live `managed-by` containers (a configured but
      non-running model never appears).
- [ ] Each row shows name, port, container status, base URL
      `http://<bind_address>:<port>/v1`, and readiness.
- [ ] Readiness reflects `/health`: `loading` while the model loads, `ready`
      once it serves; a slow/unreachable health endpoint yields `loading`
      rather than hanging the command.
- [ ] Empty state (no managed containers) renders cleanly, exit 0.
- [ ] Docker daemon unreachable → exit code 4.
- [ ] Render + command tests assert table structure and the loading/ready
      branches against fake Docker and health adapters.
